SFPCC ERP User Stories — US-18 through US-22: Estimates

Version: 1.0 · April 27, 2026
Stories: US-18 through US-22 (5 stories)

Document Purpose

This document provides detailed time estimates for user stories US-18 through US-22, which represent the next phase of ClinicSync MD ERP development for SFPCC. These estimates cover five user stories totaling 130.8 hours of development, QA, and integration work.

Context & Problem Statement

What We're Solving

Current State: SFPCC has built a robust diagnostic workflow in ClinicSync MD for sleep studies and x-rays — doctors can review, interpret, and sign diagnostic reports within the ERP. However, two critical gaps remain:

  1. Manual data entry into CureMD: After signing an x-ray interpretation in ClinicSync MD, staff must manually re-enter the interpretation into the patient's CureMD chart as a provider note. This creates duplicate work and delays referring physician access to results.
  2. Limited portal access: The medical records portal currently shows only sleep studies. Referring physicians cannot access x-ray interpretations, and newer diagnostic types (PST, PFT) have no portal distribution at all.

What US-18 through US-22 Delivers:

  • US-18: Automates x-ray interpretation insertion into CureMD provider notes — eliminates manual re-entry
  • US-19: Extends the medical records portal to support multiple document types (x-rays, sleep studies, PST, PFT) with filtering and categorization
  • US-20: Adds sleep study reports to the portal (leverages US-19 infrastructure)
  • US-21: Builds the full PST (Pulmonary Stress Test) module — order creation, report upload, doctor interpretation, automated distribution to CureMD and portal
  • US-22: Builds the full PFT (Pulmonary Function Test) module — same workflow as PST, more complex interpretation form

Business Impact: Referring physicians get faster access to diagnostic results, SFPCC staff eliminate duplicate data entry, and the ERP becomes a complete diagnostic hub for all pulmonary and imaging workflows.

X-Ray Integration Module

US-18: X-Ray Interpretation → CureMD Provider Note Integration

As a medical records user I want the signed x-ray interpretation to automatically populate a provider note in CureMD so that referring physicians can view the result in the patient's chart without manual data entry.
What it is: Integration to push signed x-ray interpretations from ClinicSync ERP into the patient's CureMD chart as a provider note. Two possible implementation paths: (1) direct CureMD API for note creation, or (2) Playwright bot to replicate the doctor's manual workflow. Requires discovery session with SFPCC to inspect CureMD note UI and identify field selectors.
Client Context

SFPCC confirmed they want x-ray interpretations visible inside CureMD charts for referring physicians, mirroring how sleep study notes already work today. The client is open to either API-based insertion or bot-based automation — whichever is more reliable. Dependency: CureMD UI discovery session (scheduled) must happen before implementation path can be finalized and bot selectors can be confirmed (if bot approach is chosen).

Open Question (Blocks Start)

OQ-1: CureMD UI discovery session with SFPCC is required to determine: (a) whether CureMD API supports note creation for this account type, or (b) which UI selectors the Playwright bot must target. Discovery session scheduled for next sprint. Bot development cannot start until selectors are confirmed.

PST Module (Pulmonary Stress Test)

US-21: PST Module — Pulmonary Stress Test Workflow PST

As a medical records user I want PST reports managed in ClinicSync MD and available to referring physicians so that pulmonary stress test results are documented and distributed automatically.
What it is: Full new PST module with 4-step workflow: (1) technician creates a PST order and uploads the raw PST report PDF, (2) doctor opens the PST review screen to see the uploaded report on the left, (3) doctor completes the interpretation form (structured checkboxes — 4 clinical fields for desaturation severity levels + free text notes) and signs, (4) signed PST document is automatically sent to CureMD and to the medical records portal. Phase 1 scope is checkbox-based interpretation only. Phase 2 (future) can add AI voice/PDF analysis at ~20–25% of original effort, following the sleep study pattern.
Client Context

SFPCC performs Pulmonary Stress Tests with no ERP module today. The client confirmed the workflow mirrors sleep study and x-ray: technician uploads the raw PST report, doctor reviews it and clicks interpretation checkboxes, document goes to CureMD and to the medical records portal. The PST interpretation form (sourced from the client's handwritten form) has four desaturation severity categories (None / Mild / Moderate / Severe) as boolean checkboxes plus a free text field for additional notes. This is the simplest new module in terms of clinical fields, but since the architecture is fully established from sleep study and x-ray, the scaffolding effort is identical.

US-22: PFT Module — Pulmonary Function Test Workflow PFT

As a medical records user I want PFT reports managed in ClinicSync MD and available to referring physicians so that pulmonary diagnostic results are documented and distributed automatically.
What it is: Full new PFT module — identical 4-step workflow to PST (US-21), same codebase pattern, different interpretation form. Interpretation form: 5 sections (Normal + 4 disease categories with Mild/Moderate/Severe + Reversibility + Diffusion Impairment) = 21 clinical fields + free text.
Create Order → Upload PFT Report → Interpret & Sign → CureMD + Portal
Client Context

SFPCC performs Pulmonary Function Tests with no ERP module today. The client confirmed the workflow mirrors PST and sleep study: technician creates an order and uploads the PFT report; the doctor reviews it on the left and checks interpretation boxes on the right; the document goes to CureMD and to the medical records portal. Phase 1 scope is checkboxes only. The PFT form (sourced from the client's handwritten form) has five distinct clinical sections — Normal, Small Airway, Obstructive Lung Disease, Restrictive Lung Disease, Restrictive Lung Disease of Chest Bellows Type, and Diffusion Impairment — with Mild/Moderate/Severe severity options and Yes/No Reversibility for the airway and lung disease sections. This is the most clinically detailed new module, but since the architecture is fully established from sleep study, x-ray, and PST, the incremental effort matches PST. Key open question: Does the Chest Bellows section also include a Reversibility field? Must be confirmed with SFPCC before DB migrations are written.

Blocking Open Question

OQ-1 (blocks migrations): Does the "Restrictive Lung Disease of Chest Bellows Type" section include a Reversibility (Yes/No) field? The handwritten form was cut off at that section. Must be confirmed with SFPCC before pft_interpretations migration is written.
