PRD: US-29-PRD-ORDER-LOCATION-FACILITY.md
Reference UI: Sleep Study Create Order — /sleep-study/create

Executive Summary

US-29 is a simple task: on each Create Order form (Sleep Study, X-Ray, PFT, PST), load all locations from Resource Planning and show them directly in a dropdown — same idea as Rendering Physician. Technician picks a location (optional); value is saved on the order for ops/accounting.

No search, no smart defaults, no appointment-based filtering. Fetch location list → display in dropdown → save locationId on create. Not shown on PDFs or Medical Records portal.

User Story

As a technician creating a clinical order,
I want to optionally select which facility (location) the patient belongs to,
So that SFPCC can track which office is handling each process for operational and accounting purposes.

Business Goals

  • Capture facility attribution at order creation without extra data entry tools.
  • Reuse the Resource Planning location master — no duplicate location list.
  • Enable future ops/accounting reporting by office (data captured in v1; reports Wave 2).
  • Keep clinical outputs unchanged — zero impact on physician interpretation workflow.

Solution Flow

RP Locations → Create Order form → POST order + locationId → Order list / detail

site_id (tenant entity) remains unchanged. location_id is the SFPCC office from Resource Planning — a separate dimension for ops/accounting.

Functional Requirements

Shared Location data source

  • On page load: GET /api/resource-planning/locations → show all locations in the dropdown.
  • Reuse useLocationMasterStore — no new APIs, no location CRUD.
  • No search, filter, or auto-select — straight list of location nicknames.

Create Order UI (all four modules)

  • Add Location (facility) Autocomplete after Rendering Physician row.
  • Field is optional (confirmed with SFPCC).
  • Display option label = location nickname.
  • Submit payload includes locationId when selected.

Backend & database

  • Single migration: add nullable location_id (FK → rp_locations) and location_nickname to all four order tables.
  • Create-order APIs accept optional locationId; validate active location when provided.
  • Include location in create audit schemaJson where audit exists today.
  • List/detail APIs return location nickname for staff visibility.

Display rules

  • Order list: Location column — nickname or "—" when blank.
  • Order detail: read-only Location field; not editable after create.
  • Excluded: interpretation PDF, CureMD note, Medical Records portal.

Acceptance Criteria

  1. All locations from Resource Planning appear directly in the Location dropdown on all four Create Order pages.
  2. Technician can create an order with a location selected — value persists and shows on list and detail.
  3. Technician can create an order without a location — order succeeds; list/detail show "—".
  4. Invalid locationId via API returns 400.
  5. Signed PDFs and portal views do not include Location.
  6. Existing required fields and Rendering Physician behavior unchanged per module.

Out of Scope (v1)

  • Required Location field
  • Edit location after order creation
  • Filters, exports, or dashboards by location
  • Billing / GL mapping by location
  • Auto-default from appointment or patient record
  • New location management UI (Resource Planning)

Effort Estimate

Simple scope. Load all locations → dropdown on 4 create forms → save locationId on order. Copy-paste the Rendering Physician pattern. Locations master already exists.
